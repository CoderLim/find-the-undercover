# 《谁是卧底》微信小程序设计文档

## 概述

本项目为一个微信小程序首版，实现《谁是卧底》的发词、看牌、揭晓流程。

本版范围严格限定为：

- 选择游戏人数，范围 `3-12`
- 选择词库分类
- 开始游戏后按人数生成玩家卡片
- 每轮固定只有 `1` 个卧底
- 卧底词与平民词不同，其余所有玩家词语相同
- 点击卡片查看词语和拼音
- 再次点击关闭卡片，并标记该玩家为 `已查看`
- 所有玩家都看完后，允许揭开谜底
- 揭晓时展示平民词、卧底词、卧底是谁

明确不在本版范围内的内容：

- 投票
- 淘汰
- 回合记录
- 主持人模式
- 自定义词库
- 历史局记录
- 词对去重策略
- 防截屏、防偷窥的系统级能力

## 目标

首版目标不是完整复刻线下游戏流程，而是提供一个适合多人同场轮流看牌的小程序工具。

成功标准：

- 进入首页后能快速完成开局设置
- 玩家可以轮流安全查看自己的词语
- 看过的玩家有清晰标记
- 未全员查看前不能误触揭晓
- 揭晓后能一眼看到平民词、卧底词和卧底玩家编号

## 实现方案

采用 `单页 + 拆组件` 方案。

理由：

- 保持单页流程，操作路径最短
- 状态集中，适合首版快速落地
- 通过组件拆分避免页面文件过重
- 为后续扩展投票、淘汰、自定义词库保留结构空间

不采用多页流程，原因是当前功能范围较小，引入页面跳转会增加状态传递复杂度，但不会带来实质收益。

## 页面结构

主页面建议为 `pages/index/index`，内部按状态展示不同区块。

页面核心区块：

- `SetupPanel`
  - 选择人数
  - 选择词库类型
  - 点击开始游戏
- `PlayerCardGrid`
  - 按人数展示 `玩家 1` 至 `玩家 N`
  - 支持翻牌、闭牌、已查看标记
- `RevealPanel`
  - 全员已查看后展示揭晓按钮
  - 揭晓后展示平民词、卧底词、卧底玩家

建议补充两个轻量公共组件：

- `SectionCard`
  - 用于统一每个功能区块的视觉容器
- `GameButton`
  - 用于统一按钮样式、禁用态、主次按钮规范

## 页面状态

页面以状态机驱动显示逻辑：

- `setup`
  - 初始状态
  - 显示开局设置区域
- `playing`
  - 游戏进行中
  - 显示卡片查看区
- `revealed`
  - 已揭晓
  - 显示结果区，并冻结交互

状态转换规则：

- 点击 `开始游戏`：`setup -> playing`
- 点击 `揭开谜底`：`playing -> revealed`
- 点击 `再来一局`：`revealed -> playing`
- 点击 `重新设置`：`playing/revealed -> setup`

## 数据模型

建议使用一个页面级状态对象管理游戏数据。

### 基础字段

- `playerCount: number`
- `categoryId: string`
- `gamePhase: 'setup' | 'playing' | 'revealed'`
- `players: Player[]`
- `answer: GameAnswer | null`
- `undercoverPlayerId: number | null`

### Player 结构

```ts
type Player = {
  id: number
  label: string
  word: string
  pinyin: string
  isUndercover: boolean
  hasViewed: boolean
  isOpen: boolean
}
```

说明：

- `id` 为玩家编号，范围 `1...N`
- `label` 固定为 `玩家 X`
- `word` 与 `pinyin` 为该玩家拿到的词
- `isUndercover` 标记是否为卧底
- `hasViewed` 表示该玩家是否完成查看
- `isOpen` 表示该玩家卡片当前是否处于打开状态

### Answer 结构

```ts
type GameAnswer = {
  commonWord: string
  commonPinyin: string
  undercoverWord: string
  undercoverPinyin: string
}
```

## 词库设计

首版采用本地内置固定词库，不接入云端，不支持自定义。

首批分类：

- `mixed` 大杂烩
- `music` 音乐
- `fairy` 童话

词库建议存放在独立模块中，例如 `data/word-bank.ts` 或等价文件。

推荐结构：

```ts
type WordPair = {
  commonWord: string
  commonPinyin: string
  undercoverWord: string
  undercoverPinyin: string
}

type WordCategory = {
  id: string
  name: string
  pairs: WordPair[]
}
```

要求：

- 每组词必须显式存储中文和拼音
- 不依赖运行时拼音生成
- 每个分类至少准备足够数量的词对，保证可玩性

首版不做本轮历史去重。即使同一分类前一局出现过某组词，下一局仍允许再次抽到。

## 发词规则

每局开始时执行以下流程：

1. 根据 `categoryId` 取出该分类词对列表
2. 随机抽取 `1` 组词对
3. 在 `1...playerCount` 中随机选出 `1` 位卧底
4. 构造 `players` 数组
5. 卧底玩家拿到 `undercoverWord / undercoverPinyin`
6. 其他玩家拿到 `commonWord / commonPinyin`
7. 所有玩家初始状态均为：
   - `hasViewed = false`
   - `isOpen = false`

## 卡片交互规则

### 背面展示

卡片背面固定显示：

- `玩家 1 / 玩家 2 / 玩家 3 ...`

玩家编号是为了便于轮流看牌时识别顺序。

### 翻牌规则

- 点击未查看的背面卡片，卡片打开，显示：
  - 词语
  - 拼音
- 再次点击该卡片，卡片关闭
- 关闭后立即将该玩家标记为 `已查看`

### 已查看规则

- 已查看卡片不能再次打开
- 已查看卡片背面显示 `已查看` 标记

### 单卡打开规则

同一时刻只允许一张卡片处于打开状态。

如果已有一张卡片打开，则必须先关闭它，才能打开下一张卡片。

该规则用于减少围观和误操作风险。

## 揭晓规则

`揭开谜底` 按钮仅在所有玩家都已查看后才可点击。

可点击条件：

```ts
players.length > 0 && players.every(player => player.hasViewed)
```

未满足条件时按钮保持禁用，不提供二次确认，不提供强制揭晓入口。

揭晓后展示：

- 平民词
- 卧底词
- 卧底是 `玩家 X`

揭晓后：

- 卡片区保留现状
- 所有卡片交互冻结
- 页面视觉重点转移到结果区

## 操作入口

### 开始游戏

触发条件：

- 已选择人数
- 已选择词库分类

未满足条件时，按钮应禁用或给出明确轻提示。

### 再来一局

作用：

- 保留当前人数
- 保留当前词库分类
- 重新随机词对
- 重新随机卧底
- 重建玩家状态

适用于连续开局。

### 重新设置

作用：

- 清空当前局数据
- 返回 `setup` 状态
- 允许重选人数和词库

## 布局与展示建议

由于人数上限为 `12` 人，卡片区应支持自动换行。

布局原则：

- 优先保证卡片可点击区域足够大
- 不强求 12 张卡片一屏全部完整展示
- 手机上允许纵向滚动
- 打开的卡片与未打开卡片应有明显视觉差异
- `已查看` 标记要醒目，但不能误导为仍可点击

## 错误处理与边界

### 输入边界

- 人数必须在 `3-12`
- 词库分类必须存在
- 当分类无可用词对时，禁止开始游戏并给出提示

### 状态边界

- `setup` 状态下不可点击揭晓
- `revealed` 状态下不可继续翻牌
- `playing` 状态下如果存在打开卡片，点击其他卡片无效

### 数据边界

- 若词库模块异常为空，应给出兜底错误提示，而不是进入空白页面
- 发词失败时不应进入 `playing` 状态

## 测试重点

首版至少应覆盖以下验证点：

- `3` 人和 `12` 人都能正常开局
- 每局只有 `1` 个卧底
- 平民词对所有非卧底玩家完全一致
- 卧底词与平民词不同
- 卡片第一次点击能打开
- 卡片第二次点击能关闭并标记已查看
- 已查看卡片不能再次打开
- 同时只能打开一张卡片
- 未全员查看前揭晓按钮不可点击
- 全员查看后揭晓按钮可点击
- 揭晓内容包含平民词、卧底词、卧底玩家
- `再来一局` 会重置本局状态但保留设置
- `重新设置` 会回到初始状态

## 未来扩展空间

当前结构为后续迭代预留以下扩展能力：

- 自定义词库
- 投票与淘汰流程
- 回合记录
- 历史局记录
- 主持人模式
- 多主题 UI 风格

这些扩展不影响本版的数据模型主干，但可能需要把页面状态机进一步拆分。

## 未决项

以下项目尚未在本轮确认，默认按推荐值执行：

- 技术栈：默认使用微信小程序原生框架，而不是 Taro / uni-app
- 视觉风格：默认做简洁、轻游戏感风格
- 首页是否显示规则说明：默认显示简短说明，不单独做规则页
- 词库容量：默认每个分类提供一个可用首版数量，不在本设计文档中锁定具体条数

## 结论

本项目首版采用微信小程序原生单页架构，通过组件拆分保证代码清晰。

交互核心是：

- 先设置
- 再轮流看牌
- 全员看完后再揭晓

该方案足够覆盖当前目标，也为后续玩法扩展保留了合理余地。
